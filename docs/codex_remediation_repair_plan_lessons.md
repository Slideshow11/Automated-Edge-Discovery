# Repair Plan Lessons — rgr-319

## Context

PR #334 was the first manual repair generated by the one-task repair-plan flow
(`run_codex_remediation_loop.py --mode one-task-repair-plan`).

The corpus entry for rgr-319 (`corpus/codex-remediation-wave2-pr314-320.json`) was
classified as `FIXED_ALREADY`. The repair plan generator accepted that classification
and produced a no-op repair packet for rgr-319 — under the assumption that the bug had
already been resolved upstream.

During review, controller-path testing showed the classification was wrong: the bug
was live in production. The fix required a real code change to
`scripts/local/run_autocoder_batch.py`.

## The Bug

In `run_autocoder_batch.py`, the validate-and-normalize ordering was:

```python
# Before fix — validate called on raw tasks (null output_root causes rejection)
valid, err = validate_task_constraints(tasks)  # ← FAIL: null output_root rejected here
if not valid:
    return HOLD_TASK_PACKET_INVALID

# Normalization happened too late:
for task in tasks:
    normalized_task = _normalize_task_packet(task, ...)  # never reached
```

The fix normalizes all task packets **before** inter-task validation:

```python
# After fix — validate called on normalized tasks
normalized_tasks = []
for task in tasks:
    if task.get("output_root") is None:
        normalized_tasks.append(
            _normalize_task_packet(task, base_sha, output_root, task["task_id"])
        )
    else:
        normalized_tasks.append(task)

valid, err = validate_task_constraints(normalized_tasks)  # ← PASS: null filled in first
```

## Lessons Learned

### 1. Corpus classifications can be wrong

`FIXED_ALREADY` is a claim, not a guarantee. The repair-plan flow should treat it as
a hypothesis to verify, not a fact to accept. In rgr-319's case, the classification
reflected what the author believed at the time of writing — but the production code
still had the bug when PR #334 was raised.

**Recommendation**: Add a verification step for `FIXED_ALREADY` tasks. If the
repair-plan output shows no production code changes, verify that the "already fixed"
claim actually holds by running the controller path test against the latest main.

### 2. Helper-only tests miss real production paths

The initial regression test for rgr-319 validated normalization in isolation
(unit-test style) but did not drive the full controller path. It proved that
`_normalize_task_packet` produces a valid task packet, but it did not prove that
`run_autocoder_batch` calls `validate_task_constraints` **after** normalization (not
before).

Codex review caught this before merge with a P2 comment:
> "This test does not actually guard the normalize-before-validate ordering: it first
> proves the raw task fails validation, then manually calls _normalize_task_packet...
> Drive the controller with a null output_root task so the regression is covered."

The final test (`test_output_root_null_normalized_before_validation`) drives the full
controller path and asserts on the actual batch status returned to the caller.

**Recommendation**: Add a checklist item to the repair plan reviewer guide:
> "Does the regression test exercise the real production path end-to-end, or does it
> only test a helper function in isolation?"

For `FIXED_ALREADY` tasks that are truly fixed, a helper-only test is fine. For tasks
where the fix was misclassified, helper-only tests give false confidence.

### 3. Codex review is a valuable safety net

The P2 comment from the Codex bot caught the test weakness before merge. This is the
second time (after rgr-318) that automated review feedback improved the regression
coverage. The feedback loop: repair-plan generates → code review catches gaps → fix
improved → merge lands correctly.

**Recommendation**: Do not dismiss Codex review comments as noise. When the bot marks
a comment as P2 and suggests exercising the controller path, treat it as a signal
that the test coverage is incomplete.

### 4. The repair-plan flow surfaces misclassifications as signal

The one-task-repair-plan mode was designed to generate fix artifacts for known bugs.
It succeeded in a different way here: it generated a no-op repair plan (correct for
`FIXED_ALREADY`), which when reviewed against production code revealed the
classification was wrong. The "no-op" output was itself useful — it flagged that the
corpus entry needed reclassification.

**Recommendation**: Keep `FIXED_ALREADY` entries in the repair corpus but add a
post-generation review step: if the repair packet contains zero production file
changes, verify against current main before closing the ticket.
